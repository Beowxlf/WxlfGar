# Wulfgar
## Security and Trust Model Specification
Version: 0.1  
Phase: Phase 1 – Local Windows CLI  
Platform: Windows 10 and Windows 11  
Language: Go  

---

# 1. Purpose

This document defines:

- The security posture of Wulfgar
- The trust boundaries
- Threat assumptions
- Data handling rules
- Integrity guarantees
- Future security requirements for server integration

Wulfgar is a diagnostic artifact engine.  
It is not an offensive tool, surveillance platform, or remote control system.

Security discipline is mandatory from Phase 1.

---

# 2. Security Objectives

Wulfgar must:

1. Operate deterministically.
2. Avoid introducing new attack surface.
3. Avoid credential exposure.
4. Avoid stealth behavior.
5. Maintain artifact integrity.
6. Prevent misuse as a remote execution tool.
7. Prepare architecture for secure server communication in Phase 2.

---

# 3. Trust Boundaries

## 3.1 Local System Boundary

Wulfgar executes on a Windows host owned or administered by the MSP.

Trust assumptions:

- The executing technician has administrative authorization.
- The system is not intentionally hostile.
- The capture driver is properly installed.

Not trusted:

- External network traffic.
- Remote endpoints.
- Captured packet contents.

---

## 3.2 Network Boundary

Wulfgar does not:

- Inject traffic.
- Spoof packets.
- Modify network sessions.

Wulfgar observes and records only.

All network data must be treated as untrusted input.

---

## 3.3 Artifact Boundary

Artifacts generated by Wulfgar:

- May contain network metadata.
- May contain IP addresses and domain names.
- May contain partial packet contents.

Artifacts must be handled as potentially sensitive operational data.

---

# 4. Threat Model (Phase 1)

## 4.1 Threat: Misuse as Remote Control Tool

Risk:
A developer expands Wulfgar to execute arbitrary commands.

Mitigation:

- Hardcoded command whitelist.
- No runtime command parameters.
- No remote execution capability.
- Explicit guardrails in architecture spec.

---

## 4.2 Threat: Sensitive Data Leakage

Risk:
Captured PCAP may contain sensitive user data.

Mitigation:

- No payload parsing for credential extraction.
- No decryption of encrypted traffic.
- Documentation requiring controlled artifact storage.
- Future optional encryption for artifact bundles.

---

## 4.3 Threat: Tool Tampering

Risk:
Generated artifacts modified before escalation.

Mitigation:

- SHA256 hashes generated for all artifacts.
- hashes.txt required.
- Integrity verification during future server upload.

---

## 4.4 Threat: Resource Exhaustion

Risk:
Large captures consume disk or CPU.

Mitigation:

- Capture duration limits.
- File size limits.
- Graceful stop mechanisms.

---

## 4.5 Threat: Privilege Abuse

Risk:
Tool runs with administrative privileges.

Mitigation:

- No registry modification.
- No credential store access.
- No system configuration changes.
- No persistent service installation in Phase 1.

---

# 5. Secure Coding Requirements

All modules must:

1. Validate inputs.
2. Avoid unsafe memory handling.
3. Use standard libraries where possible.
4. Avoid hardcoded secrets.
5. Avoid logging sensitive content.

Error handling must:

- Fail safely.
- Avoid exposing system secrets.
- Avoid stack trace leakage in user-facing output.

---

# 6. Logging and Audit Controls

Wulfgar must log:

- Start time
- Interface selected
- Commands executed
- Errors encountered
- Artifact generation status

Wulfgar must not log:

- Full packet payload contents
- Credential material
- Sensitive application data

All triage actions must be recorded.

---

# 7. Data Handling Rules

## 7.1 What May Be Stored

- Packet metadata
- Protocol headers
- IP addresses
- Domain names
- Event timestamps
- System network configuration
- Triage command output

## 7.2 What Must Never Be Stored

- Passwords
- Tokens
- Credential material
- Decrypted HTTPS content
- Browser form data
- Clipboard contents
- Memory dumps

Wulfgar is not a forensic memory extraction tool.

---

# 8. Artifact Sensitivity Classification

Artifacts must be treated as:

Operationally Sensitive Data

Recommended handling:

- Store in secured ticket system
- Avoid email transmission
- Delete when no longer operationally required

Future enhancement:

- Optional encryption at rest for artifact bundles

---

# 9. Future HTTPS Security Requirements (Phase 2)

When server integration is introduced, the system must implement:

1. HTTPS encryption.
2. Mutual TLS support.
3. Enrollment token validation.
4. Certificate pinning where feasible.
5. Role-based access control.
6. Immutable audit logging.
7. Integrity verification using SHA256.

No plaintext transmission of artifacts.

---

# 10. Abuse Prevention Constraints

The following capabilities are permanently prohibited:

- Packet injection
- Raw socket crafting
- IP spoofing
- Remote shell capability
- Arbitrary command execution
- Credential harvesting
- Lateral movement functionality
- Stealth persistence

Architectural reviews must block these expansions.

---

# 11. Consent and Authorization Policy

Wulfgar may only be deployed:

- On systems managed by the MSP
- With documented administrative authorization
- For diagnostic purposes only

Unauthorized deployment is prohibited.

---

# 12. Security Review Policy

Before each major version:

1. Review module boundaries.
2. Validate whitelist enforcement.
3. Validate schema integrity.
4. Confirm no credential handling logic introduced.
5. Confirm no new network manipulation logic added.

---

# 13. Definition of Security Compliance (Phase 1)

Phase 1 is security compliant if:

1. No arbitrary command execution is possible.
2. No packet crafting exists.
3. All artifacts are hashed.
4. No sensitive credential extraction logic exists.
5. Execution is fully visible and auditable.
6. No persistence mechanisms are installed.

---

# Summary

Wulfgar is a deterministic diagnostic artifact engine operating within strict security boundaries.

It:

- Observes only.
- Executes only predefined triage.
- Produces integrity-validated artifacts.
- Avoids offensive or manipulative behavior.
- Prepares for secure HTTPS integration.

Security is not optional.
It is part of the architecture.
